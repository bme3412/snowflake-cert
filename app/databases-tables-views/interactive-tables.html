Interactive tables and interactive warehouses

Interactive tables and interactive warehouses deliver low-latency query performance for high-concurrency, interactive workloads such as real-time dashboards and data APIs. You create interactive tables with a standard warehouse, then query them through a dedicated interactive warehouse.

Introduction

An interactive table is a specialized Snowflake table optimized for low-latency, interactive queries. A interactive warehouse is a specialized warehouse tuned for those workloads and for serving high volumes of concurrent queries. Querying interactive tables through an interactive warehouse generally gives better performance than using standard tables and warehouses. Interactive warehouses use additional metadata and indexes on interactive tables and run on the latest hardware; they are designed to run continuously and do not auto-suspend.

Interactive tables are created with CTAS-style syntax and require a CLUSTER BY clause. They support a more limited set of SQL than standard tables (e.g. only INSERT OVERWRITE for DML, no UPDATE/DELETE). Interactive tables now support join queries.

Use cases and region availability

Interactive tables are best for fast, simple queries with consistent low latency: real-time dashboards, data-powered APIs, and high-concurrency serving. Ideal queries are SELECT with selective WHERE clauses and optional GROUP BY on a few dimensions. Avoid large joins and large subqueries; performance with window functions depends on data shape.

Interactive tables and interactive warehouses are available in select AWS regions only (e.g. us-east-1, us-west-2, eu-central-1, ap-northeast-1, and others—see Snowflake docs for the full list). They are not available in all clouds or regions.

Limitations

Interactive warehouses: SELECT query timeout defaults to 5 seconds (can be reduced, not increased). The warehouse does not auto-suspend; manual suspend leads to significant latency on resume. You cannot query standard tables from an interactive warehouse—use USE WAREHOUSE to switch to a standard warehouse for other tables. Multi-cluster interactive warehouses do not auto-scale (set MIN_CLUSTER_COUNT = MAX_CLUSTER_COUNT). No CALL (stored procedures) or ->> pipe operator. No replication / failover groups.

Interactive tables: No UPDATE or DELETE; only INSERT OVERWRITE. No replication, no Fail-safe (Time Travel is supported). No materialized views on interactive tables, no streams, no dynamic table with interactive base table, no RESAMPLE. No ADD COLUMN / REMOVE COLUMN; you can rename the table, set column comments, and add masking/row access/aggregation/join policies and storage lifecycle policies. Time Travel retention is inherited from schema/database/account. Query insights are not collected for interactive table queries.

Getting started

Follow this order: (1) Create an interactive table using a standard warehouse. (2) Create an interactive warehouse. (3) Resume the interactive warehouse. (4) Add the interactive table to the interactive warehouse with ALTER WAREHOUSE ... ADD TABLES (...). (5) Set the session warehouse to the interactive warehouse and query the interactive table. Until you add the table to the warehouse, queries from the interactive warehouse will raise an object-not-found error.

Creating an interactive table

Use CREATE INTERACTIVE TABLE ... CLUSTER BY (col1, ...) AS SELECT .... Run this with a standard warehouse. The CLUSTER BY clause is required; choose columns that match the WHERE clauses of your most critical queries to improve performance.

CREATE INTERACTIVE TABLE IF NOT EXISTS orders
  CLUSTER BY (id)
AS SELECT * FROM demoSource;

For auto-refresh from a source table, add TARGET_LAG = '<num> { seconds | minutes | hours | days }' (minimum 60 seconds) and WAREHOUSE = standard_warehouse_name. You cannot use both TARGET_LAG and a storage lifecycle policy on the same interactive table.

Creating an interactive warehouse

Use CREATE WAREHOUSE or CREATE OR REPLACE WAREHOUSE with the INTERACTIVE keyword. Optionally specify TABLES (table1, table2) to associate interactive tables immediately; otherwise add them later with ALTER WAREHOUSE ... ADD TABLES.

CREATE OR REPLACE INTERACTIVE WAREHOUSE interactive_demo
  TABLES (orders)
  WAREHOUSE_SIZE = 'XSMALL';

The warehouse is created in a suspended state; run ALTER WAREHOUSE interactive_demo RESUME; before querying. Interactive warehouses have no auto-suspend option.

Managing tables and querying

Add tables: ALTER WAREHOUSE interactive_demo ADD TABLES (orders, customers); — starts cache warming, which can take significant time. You can associate an interactive table with multiple interactive warehouses.

Remove tables: ALTER WAREHOUSE interactive_demo DROP TABLES (orders, customers); — detaches the tables from the warehouse; the tables are not dropped.

Querying: USE WAREHOUSE interactive_demo; then run SELECTs against your interactive tables. In an interactive warehouse you can only query interactive tables; switch to a standard warehouse to query standard or hybrid tables.

Suspend / drop: ALTER WAREHOUSE interactive_demo SUSPEND; and DROP WAREHOUSE interactive_demo;. Resuming after suspend incurs high latency while the cache warms. Dropping the warehouse removes table associations but does not drop the tables.

Performance

Query best practices: Favor selective queries (few columns, selective WHERE with IN and time bounds). Avoid SELECT * and broad time ranges (e.g. a full year) which reduce selectivity and cache effectiveness.

Data layout: Choose CLUSTER BY columns to match filter columns (e.g. sale_date if you often filter by date). See Clustering Keys & Clustered Tables in Snowflake docs.

Point lookups: Add search optimization for point-lookup queries (e.g. WHERE id = :uuid).

Warehouse size: Size by the working set (frequently queried portion of the table). Rough guide: <500 GB → XSMALL; 500 GB–1 TB → SMALL; 1–2 TB → MEDIUM; then LARGE, XLARGE, 2XLARGE, 3XLARGE for larger working sets. Multi-cluster interactive warehouses use fixed cluster count (no auto-scale). Cache warm-up after resume can take minutes to an hour depending on data volume and warehouse size.

Troubleshooting

Single query too long: Query may need more compute (e.g. complex expressions, heavy COUNT(DISTINCT)). Try a larger warehouse size.

Sudden increase in latency (high tail / P95): Often due to insufficient cache. Increase warehouse size to get a larger local SSD cache. Allow time for cache warm-up after resume or scale-out.

Queuing or low concurrency: Use a multi-cluster interactive warehouse (MIN_CLUSTER_COUNT = MAX_CLUSTER_COUNT; no auto-scale). For short, simple queries, consider raising MAX_CONCURRENCY_LEVEL on the interactive warehouse.

Benchmarking: Disable result cache (USE_CACHED_RESULT) for consistent benchmarks. Wait for cache warm-up after resume before measuring. Run standard-table and interactive-table benchmarks separately; match CLUSTER BY to WHERE predicates and avoid clustering on very high-cardinality columns (e.g. unique IDs) unless appropriate.

Lifecycle and cost

You can attach storage lifecycle policies to interactive tables for archiving or expiring rows. You cannot use both TARGET_LAG auto-refresh and a storage lifecycle policy on the same interactive table.

Cost: Interactive warehouses are billed when active (minimum billable period one hour, then one-second granularity). Resuming a suspended warehouse starts a new minimum billable period—avoid frequent suspend/resume. Interactive tables use standard storage pricing (same as standard tables); they may be larger than equivalent standard tables due to encoding and indexes. See the Snowflake service consumption table for details.
