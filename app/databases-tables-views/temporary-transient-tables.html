Working with temporary and transient tables

In addition to permanent tables, Snowflake supports temporary and transient tables for transitory data that does not need long-term retention or full Fail-safe protection.

Introduction

Besides permanent tables (the default), Snowflake supports temporary and transient tables. They are useful for data that does not need to be kept for a long time.

You cannot create hybrid tables that are temporary or transient, or create hybrid tables inside transient schemas or databases.

Temporary tables

Temporary tables store non-permanent data (e.g. ETL, session-specific data). They exist only in the session that created them and persist only until the session ends. They are not visible to other users or sessions. When the session ends, the data is purged and is not recoverable.

Snowflake also supports other temporary objects (e.g. stages) with the same session-based semantics.

Temporary tables: storage and maintenance

While a temporary table exists, its data counts toward storage charges. To avoid unexpected charges—especially with large temp tables in long-lived sessions (e.g. over 24 hours)—drop them when no longer needed or end the session.

High volumes of temporary tables can slow queries against the Information Schema COLUMNS and TABLES views. Best practices: drop temporary tables explicitly before the session ends, and have users log out of inactive sessions (see session policies).

Potential naming conflicts

Temporary tables live in a database and schema but are session-based, so they are not subject to the same uniqueness rules. You can have a temporary and a non-temporary table with the same name in the same schema. The temporary table takes precedence in the session: it hides any other table with that name, and all operations in the session affect the temporary table.

This can cause conflicts when doing DDL (e.g. creating a temp table with the same name as an existing table, or creating a permanent table with the same name as an existing temp table). Be careful when dropping a table and using Time Travel to restore it, or when using CREATE OR REPLACE (which drops then recreates).

Important: Note this behavior with Time Travel restores and CREATE OR REPLACE.

Creating a temporary table

Use the TEMPORARY (or TEMP) keyword in CREATE TABLE. Creating a temporary table does not require the CREATE TABLE privilege on the schema.

CREATE TEMPORARY TABLE mytemptable (id NUMBER, creation_date DATE);

After creation, temporary tables cannot be converted to another table type.

Transient tables

Transient tables persist until explicitly dropped and are visible to all users with the right privileges. They are like permanent tables except they have no Fail-safe period. Use them for transitory data that must outlive a session but does not need the same level of protection as permanent tables.

Transient tables: storage and clones

Transient tables contribute to storage charges like permanent tables, but there are no Fail-safe storage costs.

When you create a transient table as a clone of a permanent table, Snowflake creates a zero-copy clone (shared micro-partitions). When you add, delete, or update rows in the clone, new micro-partitions are created for the transient table. If the source permanent table is dropped, shared micro-partitions enter Fail-safe only after the transient clone is dropped (which can delay when Fail-safe bytes are billed).

Snowflake also supports transient databases and schemas. Tables created in a transient schema, and schemas in a transient database, are transient by default.

Creating transient tables, schemas, and databases

Use the TRANSIENT keyword in CREATE TABLE, CREATE SCHEMA, or CREATE DATABASE.

CREATE TRANSIENT TABLE mytranstable (id NUMBER, creation_date DATE);

After creation, transient tables cannot be converted to another table type.

Comparison of table types

Summary of persistence, cloning, Time Travel, and Fail-safe:

Type Persistence Time Travel (days) Fail-safe (days)
Temporary Remainder of session 0 or 1 (default 1) 0
Transient Until dropped 0 or 1 (default 1) 0
Permanent (Standard) Until dropped 0 or 1 (default 1) 7
Permanent (Enterprise+) Until dropped 0 to 90 (configurable) 7

Cloning: temporary and transient can be cloned to temporary or transient; permanent can be cloned to temporary, transient, or permanent.

Time Travel and Fail-safe notes

Time Travel retention can be set at create time or later. Within that period, Time Travel operations (queries, clone, restore) are available. If a permanent table has retention 0 and is dropped, it goes straight into Fail-safe.

Temporary tables can have 1-day Time Travel, but the table is purged when the session ends, so effective retention is the shorter of 24 hours or the session. Long-running Time Travel queries delay purging of temporary and transient tables until the query finishes.

Fail-safe is not configurable. Temporary and transient tables have no Fail-safe, so no extra storage beyond Time Travel. Because transient tables have no Fail-safe, they can reduce cost for large transitory tables, but data is not recoverable after the Time Travel period (e.g. after 1 day if the table is dropped or lost). Use transient tables only for data that does not need failure protection or that can be rebuilt outside Snowflake. See Data storage considerations in Snowflake docs.
