Virtual Warehouse

No Snowflake experience? Start here. We’ll explain warehouses in plain language (and a few Python analogies).

Start here

What is a virtual warehouse?

Think of a warehouse as the “engine” that runs your work in Snowflake. You turn it on when you need it, pick how big it is, and you only pay while it’s running.

Snowflake is a cloud data platform. You store data in tables (like a database) and you run queries and load data using SQL. But before any of that can happen, Snowflake needs something to actually do the work. That something is called a virtual warehouse.

If you’ve used Python: imagine a warehouse like a pool of workers. You send a task (a query or a data load), and the warehouse runs it. The bigger the warehouse, the more workers you have. When you’re done, you can turn the warehouse off so you’re not paying for workers sitting idle.

You need a warehouse for pretty much any work: running queries (like SELECT * FROM my_table), loading data into tables, and other data operations. You get to choose the warehouse’s type (e.g. standard vs. Snowpark-optimized for heavier analytics), its size (how much compute it has), and options like “turn off automatically when idle.”

You can start and stop a warehouse anytime. You can also change its size while it’s running—like adding or removing workers without stopping the job. That makes it easy to match the warehouse to what you’re doing.

Warehouse size

Size is how much “horsepower” the warehouse has. Bigger = more compute and about double the cost per hour each step. Start small and only go bigger if you need it.

When you create a warehouse, you pick a size. Size means how much compute power that warehouse has. In Snowflake, a warehouse is made of one or more clusters—think of a cluster as a group of machines working together. The size you pick is the power per cluster.

Snowflake offers named sizes: X-Small, Small, Medium, Large, X-Large, and so on up to 6X-Large. Each step up roughly doubles the compute (and the cost per hour). So “bigger” doesn’t just mean “faster”—it also means “more expensive.” For learning or light use, X-Small or Small is usually enough.

You pay for warehouse time in credits. Credits are Snowflake’s billing unit (like paying for CPU time in the cloud). The table below shows how many credits each size uses per hour and per second. This is for the “Gen1” standard warehouses (the most common type today).

Credit usage and billing

You’re charged by the second (with a 1-minute minimum each time the warehouse starts). So if the warehouse is off, you’re not paying. Turn it off when you’re done to save money.

Snowflake charges you in credits. The more compute you use and the longer the warehouse runs, the more credits you burn. The good news: billing is per second. So if your warehouse runs for 2 minutes, you pay for 2 minutes—not a full hour. There’s one catch: every time a warehouse starts, you’re charged for at least 60 seconds. So starting and stopping a lot of times can add up; it’s better to do a batch of work, then let it turn off.

As you go up in warehouse size, credits per hour double (roughly). So an X-Large costs about 16× an X-Small per hour. The table below shows real numbers for a few sizes and running times (rounded). Use it to get a feel for what a short run vs. a long run costs.

If you use a multi-cluster warehouse (more than one cluster), you pay for each cluster that’s running. For example: a 3X-Large warehouse with one cluster for one hour, then two clusters for the next hour, would be 64 + 128 = 192 credits for those two hours. Multi-cluster is a paid (Enterprise) feature; when you’re starting out you’ll usually use a single cluster.

Does a bigger warehouse load data faster?

Not always. Loading speed depends more on how many files you’re loading than on warehouse size. For normal loads, a Small or Medium warehouse is usually enough.

When you load data into Snowflake (e.g. from files in cloud storage), the warehouse does the work. It’s tempting to think “bigger warehouse = faster load,” but that’s not how it usually works. What matters more is how many files you’re loading and how big each file is. Snowflake can parallelize across files, so many small files can use more of a big warehouse—but if you’re only loading a few files, a small warehouse is often just as fast and costs less.

So: unless you’re doing a huge bulk load with hundreds or thousands of files at once, stick with Small, Medium, or Large. Going to X-Large or bigger will burn more credits and might not make the load any faster.

Does a bigger warehouse run queries faster?

For big or complex queries, usually yes—more compute means faster. For tiny, simple queries, a bigger warehouse often doesn’t help and just costs more.

A query is a request you send to Snowflake, like “get me all rows from this table” or “sum sales by region.” The warehouse is what actually runs that query. The more compute the warehouse has, the faster it can chew through large or complicated queries. So in general: bigger warehouse → faster for heavy work.

If your queries feel slow, you can try resizing the warehouse to give it more power. You can do that even while the warehouse is running. Queries that are already running won’t suddenly get more resources, but new or queued queries will use the extra capacity once it’s ready.

One caveat: for very small, simple queries (e.g. “SELECT 1” or “SELECT * FROM tiny_table LIMIT 10”), a bigger warehouse doesn’t really help. You’re already finishing in a second or two; the bottleneck isn’t compute. So don’t assume “always use the biggest warehouse.” Match the size to the kind of work you’re doing.

Auto-suspend and auto-resume

By default, Snowflake turns your warehouse off when it’s idle (so you don’t pay) and turns it back on when you run something. You usually want to leave this on.

If you left a warehouse running 24/7, you’d pay for every second even when nobody was using it. So Snowflake gives you two behaviors that are on by default: auto-suspend and auto-resume.

Auto-suspend means: after the warehouse has been idle for a set amount of time (you can configure it), Snowflake turns it off. No queries, no load—no charge. Think of it like your laptop going to sleep when you don’t use it.

Auto-resume means: when you (or your code) run something that needs that warehouse, Snowflake turns it back on automatically. You don’t have to go into the UI and start it yourself. So your next query might take a few seconds while the warehouse wakes up, but then it runs normally.

Together, these keep your costs down and your workflow simple: use the warehouse when you need it, and let it shut down when you don’t. For multi-cluster warehouses, the same idea applies to the whole warehouse (all clusters), not to each cluster individually.

What if many queries run at once?

A warehouse has a limited amount of compute. If too many (or too heavy) queries hit it at once, some will wait in a queue until there’s room. You can resize the warehouse or use multiple clusters to handle more concurrency.

Concurrency means “how many things can run at the same time.” A warehouse has a fixed amount of compute (based on its size). Each query uses some of that compute. If you send 10 heavy queries at once and the warehouse can only run 3 at a time, the other 7 will sit in a queue until the first ones finish and free up resources. So “more users or more queries” doesn’t automatically mean “faster”—it can mean “more waiting” unless you give the warehouse more capacity.

You can set timeouts: how long a query is allowed to sit in the queue before Snowflake gives up, and how long a query can run before being killed. Those are advanced settings (e.g. STATEMENT_QUEUED_TIMEOUT_IN_SECONDS and STATEMENT_TIMEOUT_IN_SECONDS). If you notice a lot of queuing, you can create a second warehouse and send some work there, or resize the current warehouse. Snowflake also offers “multi-cluster” warehouses that add more clusters automatically when demand goes up—that’s an Enterprise feature.

How does my code or session get a warehouse?

When you connect to Snowflake (e.g. with the Python connector), that connection is a “session.” The session needs a warehouse before it can run queries. You can set a default warehouse per user, in your config, or in your connection settings. You can also switch warehouses anytime with USE WAREHOUSE.

In Snowflake, a session is basically your connection: you log in (via the web UI, Python, a SQL client, etc.) and that’s one session. By default, a new session does not have a warehouse attached. So the first time you try to run a query, Snowflake needs to know which warehouse to use. If you’ve set a default warehouse, it’ll use that automatically.

You can set a default in a few places. For users: when you create or edit a user in Snowflake, you can assign them a default warehouse. Then every time that user opens a session (e.g. from the web UI), that session will use that warehouse. For code: if you use the Snowflake Python connector, JDBC, ODBC, or tools like SnowSQL, you can pass the warehouse name in your connection settings (or in a config file). So your Python script might connect with warehouse='MY_WH' and then every query in that session runs on MY_WH.

If more than one default could apply (e.g. user has a default, and your Python code also specifies a warehouse), Snowflake uses a simple order: user default is the baseline, then the client config file (if any), then whatever you pass on the command line or in the connection parameters. The more specific one wins. And at any time during a session you can run USE WAREHOUSE other_warehouse_name; to switch to a different warehouse for the rest of that session.

For notebooks (e.g. in Snowflake’s notebook feature): Snowflake can auto-provision a small warehouse for notebook workloads. It’s meant for that use only; for heavy SQL or reporting, it’s better to point to a separate warehouse you control. Keeping notebook and “normal” query workloads on different warehouses helps with cost and performance.
