Working with Warehouses

How to create, start, suspend, and resize warehouses from the web UI or SQL (and Python). Plus how to use a warehouse in a session and how to delegate management to other roles.

Start here

Introduction

Everything you do with warehouses can be done from the Snowflake web interface (Snowsight) or with SQL. Python users can use the Snowflake connector APIs. This guide walks through the main tasks: create, start, suspend, resize, and use a warehouse.

You can manage warehouses in two main ways: through the Snowflake web interface (Snowsight) or with SQL (DDL commands like CREATE WAREHOUSE and ALTER WAREHOUSE). If you use Python, the Snowflake connector also exposes APIs for creating and operating warehouses. The behavior is the same no matter which interface you use.

Creating a warehouse

Create a warehouse in Snowsight (Compute → Warehouses → + Warehouse), with SQL (CREATE WAREHOUSE), or via the Python API. You can create it in “Started” (running) or “Suspended” state; if Started, it consumes credits once provisioned and may take a bit to be fully ready.

You can create a warehouse from Snowsight, SQL, or Python. In Snowsight: open the navigation menu, go to Compute → Warehouses, then click the “+ Warehouse” (or “Warehouse”) button and fill in the name, size, and other options. In SQL, run a CREATE WAREHOUSE command. In Python, use the connector’s warehouse create API (e.g. WarehouseCollection.create).

When you create a warehouse, you choose whether it is initially Started (running) or Suspended. If you choose Started, the warehouse begins consuming credits once all of its compute resources are provisioned. Provisioning can take a little time, so the warehouse might not be fully available for queries immediately. If you choose Suspended, no compute is provisioned until you start or resume the warehouse later.

Starting or resuming a warehouse

Start/resume from Snowsight (Compute → Warehouses → warehouse → More → Resume), SQL (ALTER WAREHOUSE ... RESUME), or Python. Credits start when compute is provisioned; there’s a 60-second minimum charge each time. Queries don’t run until the warehouse is fully provisioned (or at least 50% if some resources fail).

You can start a warehouse when you create it or anytime after. Once it exists, “resuming” a suspended warehouse is the same as starting it. In Snowsight: Compute → Warehouses → click the suspended warehouse → More options → Resume. In SQL: ALTER WAREHOUSE <name> RESUME; In Python: use the warehouse resume API.

Starting or resuming usually takes only a few seconds. In rare cases it can take longer while Snowflake provisions compute. Credits: the warehouse starts consuming credits once all requested compute resources are provisioned. If some resources fail to provision, you’re only charged for what was provisioned until the rest are repaired. Snowflake doesn’t run SQL statements submitted to the warehouse until all resources are provisioned—except if some fail during startup, in which case Snowflake may start processing once at least 50% of the requested resources are up. Billing is per second with a 1-minute minimum each time the warehouse is resumed; the UI often shows usage in hourly increments.

A warehouse must be running and set as the current warehouse for your session to process the SQL you submit. See “Using a warehouse in a session” below.

Suspending a warehouse

You can suspend a warehouse anytime, even while it’s running queries. Idle compute shuts down right away; compute that’s running statements keeps going until those statements finish, then shuts down (quiesce). After everything is down, the warehouse stops consuming credits.

A running warehouse can be suspended at any time, including while it is executing SQL. Once all compute has shut down, the warehouse stops consuming credits. In Snowsight: Compute → Warehouses → click the warehouse → More options → Suspend. In SQL: ALTER WAREHOUSE <name> SUSPEND; In Python: use the warehouse suspend API.

When you suspend, Snowflake immediately shuts down any idle compute. Compute that is currently running statements is allowed to finish those statements; when they complete, that compute shuts down and the warehouse status becomes “Suspended.” That “finish then shut down” phase is sometimes called quiesce. So you don’t lose running work—you just stop new work and let the rest complete.

Resizing a warehouse

You can resize a warehouse up or down anytime, including while it’s running. Do it in Snowsight (warehouse → More → Edit) or with ALTER WAREHOUSE ... SET WAREHOUSE_SIZE = ... Resizing up helps large queries and heavy load/unload; it doesn’t affect statements already running.

You can change a warehouse’s size (X-Small, Small, Medium, etc.) at any time, including while it is running and processing statements. In Snowsight: Compute → Warehouses → click the warehouse → More options → Edit, then change the size. In SQL: ALTER WAREHOUSE <name> SET WAREHOUSE_SIZE = 'LARGE'; (or whatever size). In Python: use the create-or-alter warehouse API.

Resizing up is useful when the warehouse would benefit from more compute—for example to speed up large, complex queries or to improve performance when loading or unloading a lot of data. The next two sections describe what happens when you resize a running warehouse vs a suspended one.

Effects of resizing a running warehouse

Resizing adds or removes compute per cluster. New compute is billed when provisioned but doesn’t run statements until it’s all ready. Existing statements keep running on the old capacity; new capacity is used for queued and new statements. Removed compute is taken away only when it’s no longer running any statements. Use WAIT_FOR_COMPLETION or SHOW WAREHOUSES to confirm provisioning.

When you resize a running warehouse, Snowflake adds or removes compute in each cluster. The same credit and usage rules as for starting/suspending apply: new compute starts using credits when it’s provisioned, but it doesn’t start executing statements until all of it is provisioned (unless some fails, in which case Snowflake may use what’s available). Compute that you remove is only taken away when it’s no longer running any statements.

Resizing has no effect on statements that are already running. When you resize up, the new compute, once fully provisioned, is used only for statements that are already in the queue and for any new statements submitted to the warehouse. So current work keeps going on the old capacity; the extra capacity kicks in for waiting and future work. To wait until the new size is fully provisioned before continuing, you can add WAIT_FOR_COMPLETION = TRUE to the ALTER WAREHOUSE command. You can also run SHOW WAREHOUSES to check the warehouse state.

Effects of resizing a suspended warehouse

Resizing a suspended warehouse doesn’t provision anything right away. It just records the new size; when you next resume the warehouse, Snowflake provisions compute for the new size and the usual start/resume billing rules apply.

If the warehouse is suspended when you resize it, Snowflake does not provision any new compute immediately. It only updates the warehouse’s size setting. The next time you start or resume the warehouse, Snowflake provisions compute according to the new size, and the same credit and usage rules for starting a warehouse apply. So resizing while suspended is a way to change size without any provisioning or billing until you’re ready to run the warehouse again.

Using a warehouse in a session

To run a query or DML statement, a warehouse must be running and must be the “current” warehouse for your session. A session has only one current warehouse at a time. Set or change it with USE WAREHOUSE <name> or the Python equivalent.

To run a query or DML statement in Snowflake, two things must be true: the warehouse must be running, and it must be the current warehouse for the session where you submit the query. A session can have only one current warehouse at a time. You set or change it with the USE WAREHOUSE <warehouse_name> SQL command or the equivalent in the Python API (e.g. WarehouseResource.use_warehouse).

Once a running warehouse is set as the current warehouse for your session, any queries and DML statements you submit in that session are processed by that warehouse. In Snowsight you can see which warehouse was used for each query: check Query History or the Workspaces page. Some Snowsight features (e.g. Task Run History, Data Preview for a table) need a warehouse to run their SQL; an X-Small is usually enough. See the Considerations guide for more on choosing a warehouse for the UI.

Delegating warehouse management

By default only ACCOUNTADMIN can manage all warehouses. You can give another role the MANAGE WAREHOUSES privilege so it can suspend, resume, resize, and describe any warehouse in the account without owning it.

By default, the ACCOUNTADMIN role can alter, suspend, describe, and perform other operations on all warehouses in the account. If you want someone else (or another role) to manage warehouses without being ACCOUNTADMIN, you can grant the MANAGE WAREHOUSES privilege to a custom role. Granting MANAGE WAREHOUSES is the same as granting MODIFY, MONITOR, and OPERATE on all warehouses in the account.

Example: you might have one role that creates and owns a warehouse (e.g. create_wh_role with CREATE WAREHOUSE ON ACCOUNT), and a second role that manages all warehouses (e.g. manage_wh_role with MANAGE WAREHOUSES ON ACCOUNT). The managing role doesn’t need to own a warehouse to suspend it, resume it, resize it, or describe it. In SQL you’d create the roles, grant the privileges, then use ALTER WAREHOUSE test_wh SUSPEND/RESUME or ALTER WAREHOUSE test_wh SET WAREHOUSE_SIZE = 'SMALL' or DESC WAREHOUSE test_wh while using the manage_wh_role.

Reviewing warehouses in Snowsight

Compute → Warehouses shows a table of all warehouses (name, status, size, clusters, running/queued statements, owner, when resumed). Click a warehouse for activity graphs, details (min/max clusters, scaling policy, auto-suspend/resume), and privileges.

To view and manage warehouses in the web UI: sign in to Snowsight, then in the navigation menu go to Compute → Warehouses. You’ll see a table of warehouses. You can search, filter by status or size, and by default you’ll see: Name, Status (Started, Resuming, Suspended), Size, Clusters (with a bar and hover for active count), Running and Queued (number of SQL statements), Owner, and Resumed (when it was last resumed). You can add columns for things like QAS scale factor, Scaling Policy, Auto Resume, Auto Suspend, and Created.

When you click a warehouse name, you get more: the Warehouse Activity section shows a graph of load over time (useful for understanding why a query might be slow), and the Details section shows status, size, min/max clusters, scaling policy, running and queued tasks, auto-suspend period, auto-resume on/off, and last resumed time. The Privileges section lets you view, grant, and revoke privileges on that warehouse. You need ACCOUNTADMIN or a role with the right warehouse privileges to see and manage these details.
