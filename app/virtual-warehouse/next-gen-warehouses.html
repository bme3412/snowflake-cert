Gen2 Standard Warehouses

Snowflake’s “next generation” standard warehouse runs on faster hardware and software. Most queries can finish faster and you can do more work at once. You choose Gen1 or Gen2 when you create or alter a warehouse (SQL only for now).

Start here

Introduction

Gen2 is a newer type of standard warehouse (not Snowpark-optimized). It uses faster hardware and software so many queries run faster and concurrency can be better. You set it in SQL with GENERATION = '2' or RESOURCE_CONSTRAINT = STANDARD_GEN_2; it’s not in the Snowsight UI yet.

In the Warehouse overview we talked about standard warehouses and their sizes. Snowflake also has a Generation 2 (Gen2) version of the standard warehouse. Gen2 is built on faster hardware and software (including better delete, update, merge, and table scan behavior). In practice, many queries finish faster and you can often do more work at the same time. How much you gain depends on your workload and configuration—Snowflake recommends testing. This guide covers how to use Gen2 and what to watch for (defaults, regions, converting, billing).

What is Gen2?

Gen2 = next-generation standard warehouse only. It does not apply to Snowpark-optimized warehouses. Gen2 is not available for 5X-Large and 6X-Large sizes.

Gen2 is an updated version of Snowflake’s standard virtual warehouse. It’s still a “standard” warehouse (the same type you use for normal SQL queries and data loading), not a Snowpark-optimized warehouse. The improvements are in performance and throughput: faster hardware and software optimizations so that, in general, more queries finish faster and the warehouse can handle more concurrent work.

Gen2 applies only to standard warehouses. It does not apply to Snowpark-optimized warehouses. Also, Gen2 is not available for warehouse sizes 5X-Large (X5LARGE) and 6X-Large (X6LARGE)—those sizes use Gen1. For memory and resource semantics: the original standard warehouse (Gen1) is equivalent to STANDARD_GEN_1 in the resource constraint model, and has the same memory capacity as MEMORY_1X for Snowpark-optimized warehouses.

How to specify Gen1 vs Gen2

Use the GENERATION clause (recommended): GENERATION = '1' or GENERATION = '2'. Or use RESOURCE_CONSTRAINT: STANDARD_GEN_1 or STANDARD_GEN_2. You must use SQL—these options are not in Snowsight yet.

When you create or alter a standard warehouse, you can choose the generation in two ways. The recommended way is the GENERATION clause: GENERATION = '1' for the original standard warehouse, GENERATION = '2' for the next generation. The other way is the RESOURCE_CONSTRAINT clause: STANDARD_GEN_1 for Gen1, STANDARD_GEN_2 for Gen2. Both work in CREATE WAREHOUSE and ALTER WAREHOUSE.

Right now the GENERATION clause and the STANDARD_GEN_1 / STANDARD_GEN_2 values are not available in Snowsight. You have to set them using SQL. So to create or switch to a Gen2 warehouse, run a CREATE WAREHOUSE or ALTER WAREHOUSE command with one of these options. Remember: Gen2 is not supported for 5X-Large and 6X-Large; use another size if you want Gen2.

Defaults by region

In some regions, new organizations get Gen2 as the default for standard warehouses (e.g. AWS Oregon, AWS Frankfurt, Azure Virginia, Azure Netherlands for orgs created after June 27, 2025; other Gen2-capable regions for orgs after July 15, 2025). Elsewhere, or for older orgs, new standard warehouses default to Gen1 unless you specify Gen2.

Snowflake is rolling out Gen2 as the default for new organizations in certain regions. For accounts in AWS US West (Oregon), AWS EU (Frankfurt), Azure East US 2 (Virginia), and Azure West Europe (Netherlands), any account in a new organization created after June 27, 2025 will have standard warehouses default to Gen2. In all other regions where Gen2 is available, new organizations created after July 15, 2025 get Gen2 as the default for standard warehouses. For the exact list of regions, see Snowflake’s region availability docs.

If your region or organization doesn’t fall into those cases, and you don’t specify GENERATION or RESOURCE_CONSTRAINT when you create a standard warehouse, Snowflake creates a Gen1 standard warehouse. So depending on where and when your account was created, “create a warehouse” might give you Gen1 or Gen2 unless you set it explicitly.

Changing generations (and warehouse type)

You can switch a warehouse between Gen1 and Gen2 (or between standard Gen2 and Snowpark-optimized) with ALTER WAREHOUSE, running or suspended. If you convert Gen1 → Gen2 while the warehouse is running, existing queries keep using Gen1 and new ones use Gen2—you’re charged for both until the old queries finish. Convert while suspended to avoid that.

You can change a standard warehouse from Gen1 to Gen2 or from Gen2 to Gen1 by altering it and setting a different GENERATION or RESOURCE_CONSTRAINT. You can do this whether the warehouse is running or suspended. You can also switch between a Gen2 standard warehouse and a Snowpark-optimized warehouse by changing WAREHOUSE_TYPE and RESOURCE_CONSTRAINT (the GENERATION clause applies only to standard warehouses and can’t be used with Snowpark-optimized).

Important: if you convert Gen1 to Gen2 without suspending first, any queries already running continue on Gen1 compute until they finish, while new queries use Gen2 compute. During that time you are charged for both sets of compute. The warehouse doesn’t auto-suspend. When the old queries complete, everything runs on Gen2. So you can convert while running to maximize availability, or convert while suspended to avoid double billing. The same idea applies when converting between standard and Snowpark-optimized (or any other RESOURCE_CONSTRAINT change): in-flight queries keep their original type/constraint; new queries use the new one.

You can see the current setting in the resource_constraint column of SHOW WAREHOUSES output. This setting is not reflected in the INFORMATION_SCHEMA views for warehouses.

Region availability

Gen2 standard warehouses are available on AWS, Azure, and GCP, in most regions. Exceptions include AWS Zurich, AWS Cape Town, GCP Dammam, and Azure US Gov Virginia. If you use account replication, secondary regions must support Gen2 for Gen2 warehouses to resume after failover.

Gen2 standard warehouses are available on AWS, Azure, and GCP. They’re available in all CSP regions except a few. Currently Gen2 is not available in: AWS EU (Zurich), AWS Africa (Cape Town), GCP Middle East Central2 (Dammam), Azure US Gov Virginia (FedRAMP High Plus), and Azure US Gov Virginia. Check Snowflake’s region availability docs for the latest list.

If you use account replication and you create Gen2 warehouses, any secondary regions must also support Gen2. Otherwise those Gen2 warehouses may not be able to resume in the secondary region after a failover. Test that Gen2 warehouses can resume in your secondary regions.

Cost and billing

Credit usage for warehouses in general is described in the Warehouse overview and Considerations guides. For Gen2 specifically, credit consumption is documented in the Snowflake Service Consumption Table—check there for current Gen2 rates.

For how credits work with virtual warehouses in general (size, clusters, per-second billing, 60-second minimum), see the Warehouse overview and Considerations guides. For the actual credit consumption of Gen2 standard warehouses, Snowflake publishes the details in the Snowflake Service Consumption Table. Look up that table for your region and warehouse size to see Gen2 rates; they can differ from Gen1.

SQL examples

Create Gen2 with GENERATION = '2' or RESOURCE_CONSTRAINT = STANDARD_GEN_2. Convert with ALTER WAREHOUSE ... SET GENERATION = '2' (or SET RESOURCE_CONSTRAINT = STANDARD_GEN_2). To switch to/from Snowpark-optimized, set WAREHOUSE_TYPE and RESOURCE_CONSTRAINT together.

Create a Gen2 warehouse (default size):

CREATE OR REPLACE WAREHOUSE my_wh
  GENERATION = '2';

Create a Gen2 warehouse with a specific size:

CREATE OR REPLACE WAREHOUSE my_wh
  GENERATION = '2'
  WAREHOUSE_SIZE = SMALL;

Using RESOURCE_CONSTRAINT instead: RESOURCE_CONSTRAINT = STANDARD_GEN_2 in the same places.

Convert an existing Gen1 warehouse to Gen2:

ALTER WAREHOUSE my_wh
  SET GENERATION = '2';

Or SET RESOURCE_CONSTRAINT = STANDARD_GEN_2; The warehouse size stays the same.

Convert Gen2 standard to Snowpark-optimized: set both WAREHOUSE_TYPE and RESOURCE_CONSTRAINT. For an X-Small–sized standard warehouse, use a memory constraint compatible with Snowpark (e.g. MEMORY_1X):

ALTER WAREHOUSE my_wh
  SET WAREHOUSE_TYPE = 'SNOWPARK-OPTIMIZED' RESOURCE_CONSTRAINT = MEMORY_1X;

Convert Snowpark-optimized to Gen2 standard: set type to standard and generation to 2. Memory capacity will align to the standard Gen2 model (equivalent to MEMORY_1X for Snowpark).

ALTER WAREHOUSE my_wh
  SET WAREHOUSE_TYPE = STANDARD GENERATION = '2';
