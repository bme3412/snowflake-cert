Interactive tables and interactive warehouses

Interactive tables and interactive warehouses deliver low-latency query performance for high-concurrency, interactive workloads such as dashboards and APIs. This feature is generally available in select AWS regions.

Introduction

This topic introduces Snowflake interactive tables and interactive warehouses. They deliver low-latency query performance for high-concurrency, interactive workloads. Interactive tables now support join queries.

Region availability: This feature is generally available in select Amazon Web Services (AWS) regions only. See the Region availability section below.

Overview of interactive warehouses and interactive tables

When you run queries on interactive tables using interactive warehouses, performance is generally better than with standard Snowflake tables.

Interactive warehouse
A specialized warehouse optimized for low-latency, interactive workloads. It tunes the engine for these workloads and uses additional metadata and indexes in interactive tables to accelerate queries. Optimized to run continuously with high concurrency. All interactive warehouses run on the latest generation of hardware.
Interactive table
A specialized table optimized for low-latency, interactive queries. Best performance when queried through interactive warehouses. Interactive tables have different ingestion methods and support a more limited set of SQL than standard tables.

Use cases

Interactive tables are optimized for fast, simple queries when you need consistent low-latency responses. Use cases include real-time dashboards, data-powered APIs, and high-concurrency serving. Simple queries that work best are usually SELECT with selective WHERE clauses, optionally GROUP BY on a few dimensions. Avoid large joins and large subqueries. Performance with window functions depends on data shape.

Region availability

Interactive tables and interactive warehouses are available in these AWS regions: us-east-1 (N. Virginia), us-west-2 (Oregon), us-east-2 (Ohio), ca-central-1 (Canada Central), ap-northeast-1 (Tokyo), ap-southeast-2 (Sydney), eu-central-1 (Frankfurt), eu-west-1 (Ireland), eu-west-2 (London). See Snowflake documentation for supported cloud regions.

Limitations of interactive warehouses and interactive tables

Interactive warehouses

  • No long-running queries: SELECT query timeout defaults to 5 seconds (can reduce, cannot increase). Other commands (e.g. SHOW, INSERT OVERWRITE) are not subject to this timeout.
  • Always running by design; no auto-suspend. Manual suspend is possible but resume has significant latency.
  • You cannot query standard Snowflake tables from an interactive warehouse. Use USE WAREHOUSE to switch warehouse type for the table type.
  • Multi-cluster interactive warehouses do not auto-scale: set MIN_CLUSTER_COUNT and MAX_CLUSTER_COUNT to the same value.
  • No CALL (stored procedures) or ->> pipe operator.
  • No replication; not in failover or replication groups.

Interactive tables

  • No UPDATE/DELETE; only DML is INSERT OVERWRITE. No replication; no fail-safe. Time Travel is supported. Query insights not collected.
  • Cannot use as source for materialized view; limited ALTER TABLE (e.g. can rename, set comments, masking/row access policies, storage lifecycle; see docs for full list). No streams; cannot be base table for dynamic table; no RESAMPLE; Time Travel retention inherited from schema/database/account.

Getting started with interactive tables

  1. Create an interactive table using a standard warehouse.
  2. Create an interactive warehouse.
  3. Resume the interactive warehouse.
  4. Add the interactive table to the interactive warehouse.
  5. Query the interactive table through the interactive warehouse.

Creating an interactive table

Use CREATE INTERACTIVE TABLE with CTAS-style syntax and the required CLUSTER BY clause. Choose clustering columns to match WHERE clauses in your most time-critical queries; they significantly affect performance. Run CREATE INTERACTIVE TABLE with a standard warehouse; use the interactive warehouse only for querying later.

CREATE INTERACTIVE TABLE
  IF NOT EXISTS orders
  CLUSTER BY (id)
AS
  SELECT * FROM demoSource;

Auto-refresh

To auto-refresh from another table, use TARGET_LAG with an interval (e.g. '20 minutes') and WAREHOUSE (a standard warehouse for refresh). Minimum is 60 seconds. Example:

CREATE INTERACTIVE TABLE my_dynamic_interactive_table
  CLUSTER BY (c1, c2)
  TARGET_LAG = '20 minutes'
  WAREHOUSE = my_standard_warehouse
AS SELECT c1, SUM(c2) FROM my_source_table GROUP BY c1;

Creating an interactive warehouse

Use CREATE WAREHOUSE or CREATE OR REPLACE WAREHOUSE with the INTERACTIVE keyword. Optionally use the TABLES clause to associate interactive tables immediately.

CREATE OR REPLACE INTERACTIVE WAREHOUSE interactive_demo
  TABLES (orders)
  WAREHOUSE_SIZE = 'XSMALL';

Or create without tables and add them later with ALTER WAREHOUSE ... ADD TABLES. Interactive warehouses stay active by default (no auto-suspend).

Performance considerations

Query best practices

Interactive warehouses benefit most from selective queries (few columns, selective WHERE). Avoid SELECT * and broad time ranges (e.g. a full year). Large joins and compute-heavy expressions (e.g. regex) can reduce concurrency.

Data layout

Cluster by columns you filter on (e.g. sale_date for date-range queries). See Clustering Keys & Clustered Tables in Snowflake docs.

Search optimization

Use search optimization for point lookups (e.g. WHERE some_id = some_UUID).

Sizing the warehouse

Size based on the working set (frequently queried portion). Interactive warehouses use local storage caching; non-cached data has higher read latency on first read. Sizes XSMALL–3XLARGE; multi-cluster is supported but min and max cluster count must be equal (no auto-scale). Suggested sizes: <500 GB → XSMALL; 500 GB–1 TB → SMALL; 1–2 TB → MEDIUM; 2–4 TB → LARGE; 4–8 TB → XLARGE; 8–16 TB → 2XLARGE; >16 TB → 3XLARGE. You don’t need the entire working set in cache—hot data is enough.

Troubleshooting

Single query slow: Query may need more compute (e.g. regex, COUNT(DISTINCT)). Try a larger warehouse. Sudden high tail/P95 latency: Often insufficient cache; increase warehouse size; allow cache warm-up after resume. Queuing or low concurrency: Scale out with MIN_CLUSTER_COUNT and MAX_CLUSTER_COUNT (set equal); multi-cluster doesn’t auto-scale.

Adding and removing interactive tables

Add tables to an interactive warehouse (one-time) so you can query them; otherwise you get object not found. Use ALTER WAREHOUSE ... ADD TABLES (orders); (comma-separated for multiple). This starts cache warming, which can take significant time. You can associate one interactive table with multiple interactive warehouses.

Remove with ALTER WAREHOUSE interactive_demo DROP TABLES (orders, customers);. The tables still exist; this only detaches them from the warehouse.

Resuming and suspending an interactive warehouse

Interactive warehouses are created suspended. Resume with ALTER WAREHOUSE interactive_demo RESUME;. Queries will be slow while the cache warms (minutes to an hour depending on data). Suspend with ALTER WAREHOUSE interactive_demo SUSPEND;. In production you typically leave them running; Snowflake does not auto-suspend interactive warehouses. To remove entirely: DROP WAREHOUSE; this removes associations but interactive tables remain and can be used with other interactive warehouses.

Querying an interactive table

Set the session warehouse to your interactive warehouse: USE WAREHOUSE interactive_demo;. Then run queries against the interactive table normally. In an interactive warehouse you can only query interactive tables; switch to a standard warehouse to query standard or hybrid tables.

Benchmarking best practices

  • Turn off query result cache (e.g. USE_CACHED_RESULT session parameter) for consistent benchmark runs.
  • Wait after create/resume for cache warm-up before measuring.
  • Don’t interleave standard and interactive table queries when comparing; run full benchmark on one, then the other.
  • Match clustering columns to WHERE predicates; avoid high-cardinality columns (e.g. unique IDs) as sole clustering key.
  • For short, simple queries, consider higher MAX_CONCURRENCY_LEVEL on the interactive warehouse.

Cost and billing

Storage lifecycle policies can be used to archive or expire rows on interactive tables. You cannot use both TARGET_LAG (auto-refresh) and a storage lifecycle policy on the same interactive table.

Interactive warehouses incur compute charges when active. Minimum billable period is one hour, then one-second granularity. Resuming a paused warehouse starts a new minimum billable period—avoid pausing and resuming repeatedly in a short period.

Interactive tables use standard storage pricing (same as standard tables). They may be larger than equivalent standard tables due to encoding and indexes; that size is included in storage volume. See the Snowflake service consumption table for details.
