Deployment on AWS, Azure, GCP — Regions and Cross-Cloud Concepts

Where Snowflake runs, account placement, and replicate-then-share.

Snowflake Is Not a Cloud Provider — It Runs On Top of Them

One of the first conceptual adjustments to make when learning Snowflake is understanding what kind of product it actually is. Snowflake does not own or operate data centers. It does not have its own network backbone or physical servers. It is a cloud-native SaaS platform — software that runs entirely on top of the three major public clouds: Amazon Web Services, Microsoft Azure, and Google Cloud Platform.

This means Snowflake's storage layer is literally the cloud provider's object storage service: Amazon S3 on AWS, Azure Blob Storage on Azure, Google Cloud Storage on GCP. Snowflake manages that storage on your behalf — formatting it as compressed micro-partitions, maintaining metadata, enforcing access controls — but the physical infrastructure belongs to the cloud provider. Similarly, Snowflake's compute nodes (Virtual Warehouses) run on the cloud provider's virtual machine infrastructure. When you resume a warehouse, Snowflake is provisioning VMs from AWS, Azure, or GCP, not from Snowflake's own hardware pool.

Snowflake's value is entirely in the software layer it builds on top of this infrastructure: the three-layer architecture, the query optimizer, the global metadata store, the security and governance model, the result cache, the automatic micro-partition management. None of those things require owning the hardware underneath. And the fact that Snowflake doesn't own hardware is what gives it global reach — anywhere AWS, Azure, or GCP operate, Snowflake can operate too, without Snowflake having to build or maintain any physical presence there.

This architecture also explains something about Snowflake's pricing model. Storage costs in Snowflake closely track the underlying cloud provider's object storage costs — Snowflake is essentially re-selling that storage with its own compression and management layer on top. Compute costs track the underlying VM costs. When cloud providers reduce their infrastructure pricing, some of that can flow through to Snowflake customers.

How Snowflake Accounts Are Deployed

Every Snowflake account is created in a specific cloud provider and region. When you sign up for Snowflake or when your organization creates an account, you select both the cloud (AWS, Azure, or GCP) and the specific geographic region within that cloud (for example, AWS US East - N. Virginia, or Azure West Europe - Netherlands). This selection is permanent for the lifetime of that account. You cannot migrate a Snowflake account from one cloud provider to another. You cannot move it from one region to another. The account, its storage, and its compute all live in the selected cloud and region indefinitely.

The region selection has two significant practical implications. The first is data residency. The physical location of your data is determined entirely by your account's region. If you create an account in AWS US East (N. Virginia), your micro-partitions are stored in AWS's N. Virginia data centers. If you create an account in Azure West Europe (Netherlands), your data is in Microsoft's Dutch facilities. For organizations operating under regulations that require data to remain within a specific country or jurisdiction — GDPR in Europe, data sovereignty laws in various national contexts — the account region is the only Snowflake-level control that governs where data physically resides. Selecting the wrong region at account creation puts you in compliance violation from the moment you load your first byte of regulated data. There is no reconfiguration path after the fact.

The second implication is network latency. Queries originate from a client — a BI tool, a Python script, a data pipeline — and travel to Snowflake's endpoint in the account's region. The closer the client is to the Snowflake account's region, the lower the round-trip latency. For most analytical workloads where query execution time is measured in seconds or minutes, network latency is a minor concern — 50 extra milliseconds of network round-trip is negligible against a 30-second query. But for high-frequency, low-latency operational workloads — applications that issue many short queries per second — region proximity can meaningfully affect end-to-end response time.

Available Regions Across the Three Clouds

Snowflake supports multiple regions across all three cloud providers, spanning North America, South America, Europe, Asia Pacific, and other geographies. The specific list changes over time as Snowflake expands to new regions, so the exam does not test region name memorization. You will not be asked "which AWS region hosts Snowflake in the US East?" — the exam tests concepts, not geography trivia.

What the exam does test is the implications of region selection and region boundaries. Three principles govern this:

Data does not automatically cross regional boundaries. When you load data into a Snowflake account in a given region, that data stays in that region. Snowflake does not replicate it elsewhere, does not cache it in other regions, and does not route queries through other regions. The data's physical location is determined by the account region and stays there unless you explicitly and intentionally move it using replication. This is the foundation of data residency compliance: your GDPR-covered data in an EU Snowflake account does not leave the EU unless you deliberately replicate it somewhere else.

Not every cloud provider region has a Snowflake deployment. Snowflake deploys in a subset of AWS, Azure, and GCP regions based on demand and strategic priorities. Before selecting a cloud and region for a Snowflake account, you should verify it is supported in the Snowflake documentation. The exam won't ask you to identify unsupported regions, but it's conceptually important to know the list isn't exhaustive.

Cross-region communication adds latency and egress cost. When data physically moves from one region to another — whether through replication, cross-region query federation, or cross-region sharing — it incurs both network latency (the transfer takes time) and cloud egress costs (cloud providers charge for data leaving their network). Same-region data movement within Snowflake has neither of these costs. This is why Snowflake's default posture is to keep data local to the account region, and cross-region movement is always opt-in and deliberate.

The Account Identifier

Every Snowflake account has a unique identifier used in connection strings, URLs, and cross-account operations. There are two formats, and you need to understand both for the exam.

The original format is the account locator. This is a short alphanumeric string automatically assigned to an account at creation — something like xy12345. When used in a fully qualified connection string, it becomes xy12345.us-east-1.aws, combining the locator with the region identifier and cloud provider. The account locator is unique within its region but not globally — two accounts in different regions could theoretically have the same locator string. For this reason, the locator alone is insufficient to globally address an account; you always need the region and cloud qualifier for it to be unambiguous.

The modern format is the organization-based account identifier, in the form <orgname>.<account_name>. The organization name is the name of your Snowflake organization (a multi-account administrative construct), and the account name is a human-readable name assigned at account creation. This format is globally unique across all of Snowflake — no two accounts anywhere share the same org name and account name combination. It is also more readable than a cryptic locator string. The organization-based identifier is the preferred format for modern Snowflake deployments, particularly for replication configuration, Snowflake Private Connectivity, and any cross-account features where a globally unambiguous identifier is required.

Both formats appear in connection strings, SHOW output, and documentation. The exam distinguishes between them by testing which one is globally unique (org-based), which was the original format (locator), and in which contexts each is appropriate.

Cross-Cloud and Cross-Region Concepts

Because a Snowflake account is permanently bound to one cloud and region, any scenario involving multiple geographies or multiple cloud providers requires deliberate mechanisms to bridge those boundaries. Snowflake provides two main paths: replication and sharing. Understanding exactly what each can and cannot do, and what edition each requires, is one of the most directly tested areas of the exam.

Database replication

Database replication allows you to create a synchronized copy of a Snowflake database in a different account — which can be in a different region, a different cloud provider, or both. The source account holds the primary database; target accounts hold secondary databases that are read-only replicas. Snowflake manages the replication process asynchronously — the secondary is eventually consistent with the primary, meaning there is a lag between when data changes in the primary and when those changes appear in the secondary. The size of this lag depends on replication frequency configuration.

Database replication requires Enterprise edition or above. A Standard edition account cannot replicate its databases to another account.

Replication Groups extend the concept by allowing you to bundle multiple databases, shares, and other account objects together as a single coordinated replication unit — ensuring consistency across related objects rather than replicating each database independently. Failover Groups add automated failover orchestration on top of Replication Groups, creating a path to promote a secondary account to primary status in the event of a regional outage. Both Replication Groups and Failover Groups require Business Critical edition or above.

Secure Data Sharing and the same-region constraint

Secure Data Sharing is Snowflake's zero-copy data sharing mechanism: a provider account grants a consumer account read access to specific database objects. No data is copied or moved — the consumer's Virtual Warehouses simply query the provider's storage directly. This is elegant and efficient, but it has a hard architectural constraint: it only works natively between accounts in the same region on the same cloud provider.

The reason is physical. Zero-copy sharing works because the consumer's compute can reach the provider's storage directly — they are in the same physical infrastructure environment. If the consumer's account is in a different region or a different cloud, their compute cannot reach the provider's storage by the same low-latency, zero-cost path. Cross-region or cross-cloud sharing would require data to physically travel across regional or cloud boundaries, which Snowflake does not do natively in a sharing context.

The solution is the replicate-then-share pattern. To share data with a consumer in a different region or on a different cloud, the provider first replicates the relevant data to an account that exists in the same region and cloud as the consumer, then shares from that replicated account. The consumer gets the same read-only access experience — they query through their own warehouse — but the provider maintains an intermediate replicated account as the sharing point. This pattern adds replication lag, intermediate account maintenance overhead, and replication costs, but it is the supported and correct architectural answer to cross-boundary sharing.

The Snowflake Marketplace presents a simplified face over this complexity for published data products. Marketplace listings appear available globally regardless of the consumer's region or cloud, because Snowflake manages the replication and distribution infrastructure in the background. From the consumer's perspective, a Marketplace dataset simply appears in their account.

The Organizational Layer

Above the individual account sits the Organization — a Snowflake construct that allows a company to manage multiple Snowflake accounts under a single administrative umbrella. An organization might contain separate accounts for production, development, and staging environments; accounts in different cloud regions serving different business units; or accounts on different cloud providers to serve different geographic markets.

The ORGADMIN role sits at the top of the role hierarchy above ACCOUNTADMIN. From the ORGADMIN context, you can view credit usage and compute consumption across all accounts in the organization, manage account creation, configure cross-account replication, and oversee the organization's overall Snowflake footprint. ORGADMIN is the role that gives multi-account enterprises a unified administrative view of their entire Snowflake deployment — something that would otherwise require logging into each account individually.

The organization is also the namespace that makes organization-based account identifiers globally unique. The format <orgname>.<account_name> is unique because the organization name itself is globally unique across Snowflake. Two different companies cannot share the same organization name. Within an organization, account names must be unique. Together, org name plus account name form an address that unambiguously identifies any account anywhere in Snowflake's global infrastructure, regardless of which cloud or region it's deployed in.

For the exam, the key facts about the organization layer are: ORGADMIN has cross-account visibility; organizations are the administrative grouping above accounts; and organization-based identifiers are the globally-unique, modern way to address accounts in cross-account operations.

The Exam Angle

Cloud deployment questions on the SnowPro Core exam tend to revolve around a small set of recurring scenario types. Recognizing the pattern immediately narrows the answer choices.

Exam scenario Key concept Correct answer direction
"Company needs data in EU for compliance but already has a US Snowflake account" Account placement is permanent Create a new Snowflake account in an EU region; cannot migrate the existing account
"Provider in us-east-1/AWS wants to share data with consumer in eu-west-1/AWS" Sharing is same-region, same-cloud only Replicate to an account in eu-west-1/AWS, then share from that account
"Provider in AWS wants to share data with consumer on Azure" Cross-cloud sharing requires replicate-then-share Replicate to an account on Azure in the consumer's region, then share
"Which edition is required to replicate a database across regions?" Edition requirements for replication Enterprise edition or above
"Which feature provides orchestrated disaster recovery with automated failover?" Edition requirements for DR Failover Groups — requires Business Critical or above
"What uniquely identifies a Snowflake account globally?" Account identifier formats The organization-based identifier (<orgname>.<account_name>), not the account locator
"Who can manage and view usage across all accounts in an organization?" ORGADMIN role A user with the ORGADMIN role granted in any account of the organization
The most common distractor: Exam questions sometimes imply that cross-region sharing can be solved purely by a configuration option or a sharing flag — something quick. The answer is always no: the provider must establish a second account in the consumer's region, replicate there, then share from that account. No shortcut exists. If an answer choice suggests otherwise, it is wrong.
